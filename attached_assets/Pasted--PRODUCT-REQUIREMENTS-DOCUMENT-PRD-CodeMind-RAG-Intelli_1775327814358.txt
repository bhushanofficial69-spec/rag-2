## EXECUTIVE SUMMARY

**CodeMind RAG** is a web-based "Google for Code" that enables developers to ask natural language questions about their codebase and receive instant, accurate answers with exact file references, code snippets, and dependency impact warnings.

### Problem Statement

Developers waste **5-10 hours per week** searching through codebases to answer questions like:
- "How does user authentication work?"
- "Where is the payment validation logic?"
- "If I change this function, what breaks?"

Current solutions fall short:
- **GitHub Copilot:** Explains only single files, not codebase patterns
- **grep/IDE search:** Finds exact matches only, misses semantic relationships
- **Documentation:** Often outdated or incomplete
- **Asking seniors:** Bottleneck, doesn't scale

### Solution Overview

CodeMind RAG combines **Retrieval-Augmented Generation (RAG)** + **Hybrid Search** to understand both semantic meaning (vector search) and exact terms (keyword search) across entire codebases. It answers questions in <3 seconds with precise citations.

### Target Impact

- **Developers:** 50% faster onboarding, instant codebase answers
- **Teams:** Reduce knowledge silos, improve code documentation
- **Portfolio Value:** Demonstrate advanced AI/ML system design

#### Core Features
1. **GitHub Repo Ingestion**
   - Clone public GitHub repos
   - Filter code files (.py, .js, .ts, .java)

   - Skip dependencies (node_modules, .git, __pycache__)

   - Dense vector search (semantic)
   - Sparse BM25 search (keyword)
   - Reciprocal Rank Fusion merging
   - Top-10 relevant chunks returned

   - Chat interface (question input)

## DETAILED REQUIREMENTS

### Functional Requirements

#### FR-1: Repository Ingestion

**Description:** System can ingest code from GitHub repositories and prepare for search.

**Requirements:**
- FR-1.1: Accept public GitHub repository URLs (format: https://github.com/owner/repo)
- FR-1.2: Support main and develop branches (user-selectable)
- FR-1.3: Clone repository securely to temporary storage
- FR-1.4: Filter files by extension (.py, .js, .ts, .java)
- FR-1.5: Exclude directories (node_modules, .git, __pycache__, dist, build, vendor)
- FR-1.6: Skip files >100KB
- FR-1.7: Process max 5000 files per repo
- FR-1.8: Clean up temporary files after processing
- FR-1.9: Track ingestion progress (files indexed, chunks created)
- FR-1.10: Handle errors gracefully (invalid URL, connection timeout, permission denied)

**Acceptance Criteria:**
- ✓ Can ingest a 1000-file Python repository in <10 minutes
- ✓ Progress endpoint shows accurate file count and status
- ✓ Temporary files deleted after completion
- ✓ Error messages are user-friendly and actionable

---

#### FR-2: Language-Aware Code Chunking

**Description:** Code is split intelligently, respecting language structure (functions, classes).

**Requirements:**
- FR-2.1: Detect language from file extension
- FR-2.2: Use language-specific separators (Python: \nclass, \ndef; JS: \nfunction, \nconst, etc.)
- FR-2.3: Target chunk size of 512 tokens with 50-token overlap
- FR-2.4: Preserve complete function/class definitions (don't split mid-function)
- FR-2.5: Extract line numbers for each chunk (start_line, end_line)
- FR-2.6: Extract function names from code
- FR-2.7: Skip chunks <100 tokens
- FR-2.8: Handle edge cases (multi-line strings, comments)

**Acceptance Criteria:**
- ✓ No function splits mid-definition
- ✓ Line numbers are accurate (±1 line)
- ✓ Chunks average 400-512 tokens
- ✓ Function names correctly extracted for 95%+ of Python functions

---

#### FR-3: Embedding Generation

**Description:** Code chunks are converted to vector embeddings for semantic search.

**Requirements:**
- FR-3.1: Use sentence-transformers/all-MiniLM-L6-v2 model
- FR-3.2: Generate 384-dimensional dense vectors
- FR-3.3: Batch process chunks (16 per batch for efficiency)
- FR-3.4: Cache embeddings to avoid regeneration
- FR-3.5: Handle HuggingFace API rate limits (retry with backoff)
- FR-3.6: Timeout: 30 seconds per batch
- FR-3.7: Track embedding generation metrics

**Acceptance Criteria:**
- ✓ All 1500+ chunks from test repo have embeddings
- ✓ Embedding dimensions are exactly 384
- ✓ Duplicate chunks return cached embeddings (no API call)
- ✓ Batch processing reduces API calls by 16x vs single requests

---

#### FR-4: Qdrant Vector Database Integration

**Description:** Code chunks with embeddings are stored in Qdrant Cloud for retrieval.

**Requirements:**
- FR-4.1: Create Qdrant collection with dense vectors (384 dims, COSINE distance)
- FR-4.2: Create sparse vectors with BM25 weighting (IDF modifier)
- FR-4.3: Store payload metadata: file_path, start_line, end_line, language, function_name, code_content, dependencies
- FR-4.4: Support upsert (update existing chunks)
- FR-4.5: Handle connection errors gracefully
- FR-4.6: Implement connection pooling
- FR-4.7: Monitor collection size (warn at 80% of 1GB limit)

**Acceptance Criteria:**
- ✓ 1500 chunks stored with metadata in Qdrant
- ✓ Connection errors logged and retried
- ✓ Can verify chunks in Qdrant dashboard
- ✓ Collection stays under 1GB limit

---

#### FR-5: Hybrid Search

**Description:** Search combines semantic understanding with keyword matching using Reciprocal Rank Fusion.

**Requirements:**
- FR-5.1: Convert user question to embedding
- FR-5.2: Execute dense vector search (top-10 results)
- FR-5.3: Execute sparse BM25 search (top-10 results)
- FR-5.4: Merge results using Reciprocal Rank Fusion (k=60)
- FR-5.5: Return top-10 ranked results
- FR-5.6: Include RRF score and confidence for each result
- FR-5.7: Query latency <1 second
- FR-5.8: Handle empty result sets gracefully

**Acceptance Criteria:**
- ✓ Hybrid search outperforms dense-only on exact terms (e.g., "Error 503")
- ✓ Hybrid search outperforms sparse-only on synonyms (e.g., "login" vs "authentication")
- ✓ Query latency <1 second for 1500-chunk collection
- ✓ No results returns helpful "not found" message

---

#### FR-6: LLM-Based Answer Generation

**Description:** Retrieved code chunks are sent to Groq LLM to generate natural language answers.

**Requirements:**
- FR-6.1: Format code context into readable markdown
- FR-6.2: Use strict prompt template to prevent hallucinations
- FR-6.3: Enforce citations (file path + line numbers)
- FR-6.4: Limit answer length to 300 words
- FR-6.5: Include code snippets (max 10 lines each) in answer
- FR-6.6: Use Groq llama-3.1-70b-versatile model
- FR-6.7: Temperature set to 0.3 (deterministic)
- FR-6.8: Timeout: 30 seconds
- FR-6.9: Retry failed LLM calls up to 3 times

**Acceptance Criteria:**
- ✓ 95%+ of answers include valid file citations
- ✓ No made-up code or functions in answers
- ✓ Code snippets are accurate (copy-paste from source)
- ✓ LLM errors are logged with context

---

#### FR-7: Citation Extraction

**Description:** File paths and line numbers are automatically extracted from LLM responses and linked.

**Requirements:**

- FR-7.2: Extract line ranges: `(lines 45-78)` or `(line 45)`
- FR-7.3: Validate citations against retrieved chunks
- FR-7.4: Return structured citation data
- FR-7.5: Handle malformed citations gracefully
- FR-7.6: Calculate confidence score (% of answer text that's cited)

**Acceptance Criteria:**
- ✓ 90%+ of citations are valid and found in source
- ✓ Line numbers match actual code locations (±1 line acceptable)
- ✓ Answer has >80% citation coverage

**Acceptance Criteria:**

## DATA REQUIREMENTS

### Data Models

#### Code Chunk (Stored in Qdrant)

```json
{
    "id": "uuid-123-abc",
    "file_path": "src/auth/login.py",
    "start_line": 45,
    "end_line": 78,
    "language": "python",
    "function_name": "validate_user",
    "code_content": "def validate_user(username, password):\n    hash = bcrypt.hash(...)",
    "repo_name": "user/my-project",
    "dependencies": [
        "src/api/users.py",
        "src/middleware/auth.py"
    ],
    "external_dependencies": [
        "bcrypt",
        "sqlalchemy"
    ],
    "last_updated": "2026-04-04T10:00:00Z",
    "chunk_size_tokens": 512,
    "vector_dense": [0.123, 0.456, ...],  // 384 dims
    "vector_sparse": {                     // BM25 weights
        "def": 2.5,
        "validate": 1.8,
        "user": 1.2,
        ...
    }
}
```

#### Query Request

```json
{
    "question": "How does user authentication work?",
    "repo_name": "user/my-project",
    "top_k": 10
}
```

#### Query Response

```json
{
    "question": "How does user authentication work?",
    "answer": "User authentication happens in 3 places...",
    "sources": [
        {
            "file_path": "src/auth/login.py",
            "start_line": 45,
            "end_line": 78,
            "language": "python",
            "snippet": "def validate_user(username, password):",
            "rrf_score": 0.92
        }
    ],
    "confidence_score": 0.89,
    "query_time_ms": 2345,
    "tokens_used": {
        "question_embedding": 12,
        "llm_input": 342,
        "llm_output": 156
    }
}
```

### Data Storage

| Data | Storage | Retention | Backup |
|------|---------|-----------|--------|

| Ingestion jobs | In-memory (MVP) | 24 hours | None (lose on restart) |
| Query metrics | None (MVP) | - | - |
| Code from GitHub | Temp storage | During ingestion | Auto-delete |

## GLOSSARY

| Term | Definition |
|------|-----------|
| **RAG** | Retrieval-Augmented Generation - fetch relevant context, send to LLM for answer |
| **Hybrid Search** | Combines semantic (vector) + keyword (BM25) search |
| **RRF** | Reciprocal Rank Fusion - algorithm to merge ranked lists |
| **Dense Vectors** | 384-dimensional embeddings from sentence-transformers model |
| **Sparse Vectors** | BM25 weighted keyword scores (high for rare words) |
| **LLM** | Large Language Model (Groq llama-3.1-70b-versatile) |
| **Chunking** | Splitting code into 512-token pieces |
| **Embedding** | Vector representation of text (captures meaning) |
| **Citation** | File path + line numbers proving answer is from code |
| **Dependency** | When one file imports or calls another |
| **Transitive Dependency** | Indirect dependency (A→B→C means A depends on C indirectly) |
